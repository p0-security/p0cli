## Purpose

The `p0` CLI sits on the critical path of how engineers obtain just-in-time access to production infrastructure. A regression here doesn't just annoy a user â€” it can block an on-call engineer from reaching a box during an incident, leak a credential to the wrong stream, or silently clobber a `~/.kube/config` / `~/.aws` profile. This document exists to **drive down bugs and regressions** by enumerating, exhaustively, the *observable behaviors* an end-to-end test suite must lock down.

It catalogs **330+ test cases across 15 sections** â€” one per command/plugin subsystem, plus a cross-cutting section that audits the whole surface for error-handling, output-stream, signal, platform, argument-parsing, and secret-safety invariants. Each case is written to be **observable and deterministic**: it asserts on stdout, stderr, exit code, files/permissions written, temp-file cleanup, subprocess argv, or prompts â€” never on internal implementation details.

This is a *specification of intent*, not a claim of current coverage. Today the repo has unit/integration tests under `src/**/__tests__`, but **no harness that drives a full `p0 <args>` invocation end to end**; the "Coverage notes" line under each section flags where today's tests stop short.

### The e2e test model used here

An e2e case **invokes the CLI the way a user does** â€” through its yargs entry (`getCli().parse(argv)` or a command registrar such as `sshCommand(yargs()).parse("ssh host â€¦")`) â€” and **mocks only the external boundaries**:

- **P0 API** â€” `src/drivers/api.ts` (`fetchCommand`, `fetchStreamingStatus`, `fetchOrgData`, `fetchIntegrationConfig`, `authFetch`, â€¦)
- **Firebase / Firestore** â€” `src/drivers/firestore.ts`, the `firebase` SDK
- **Subprocesses** â€” `src/common/subprocess.ts` (`ssh`, `scp`, `aws`, `kubectl`, `az`, `gcloud`, `session-manager-plugin`, `curl`, `nc`/`ncat`, `claude`)
- **Filesystem** â€” config / identity / cache / key / `~/.aws` / `~/.kube` files (prefer a temp `HOME`/`P0_DIR` over `fs` stubs where possible)
- **Browser open**, the **local auth HTTP callback server**, and **S3** (`@aws-sdk/*`)
- **TTY / shell / platform** â€” `sys.writeOutputIsTTY()`, `SHELL`, `getOperatingSystem()`, `isSea()`

Everything between the argv and those boundaries â€” parsing, validation, control flow, output routing, file writes, subprocess argv construction, cleanup â€” is the **system under test**.

## Test harness recommendations

The existing tests already establish the right primitives (`vi.mock("../../drivers/api")`, `vi.mock("../../drivers/auth")`, the `print1`/`print2` spies in `src/commands/__tests__/ssh.test.ts`, the `failure()` helper in `src/testing/yargs.ts`, `mockAuthn` in `src/testing/authn.ts`). To scale them into a reliable e2e suite:

1. **A single `runCli(argv: string)` harness.** Wrap "build a fresh yargs instance for the full command set, parse an argv string, capture result." Return `{ stdout, stderr, exitCode, error }` by collecting `print1`/`print2` calls and intercepting `exitProcess`/`process.exit`. This removes per-test boilerplate and makes every case a one-liner against the same entry as production.
2. **Assert stdout and stderr separately, always.** The single highest-value invariant in this CLI is stream discipline (eval contract, bearer token, ssh-proxy/resolve, MCP, `ls --json`). Provide `expectStdout()` / `expectStderr()` helpers and default to asserting stdout is *empty* unless a case explicitly expects a payload.
3. **Centralized fixtures & mock factories.** One module that builds: a logged-in `Authn`, an org "discover doc" (Firebase, ssoPassthrough, custom help/contact), an integration config (aws/idc/okta/gcloud/azure/eks installed states), STS/SAML/IDC responses (reuse `src/commands/aws/__tests__/__input__/`), and streaming-status generators (NEWâ†’DONE / DENIED / EXPIRED / ERRORED). Parameterize providers so one test body covers aws/gcloud/azure/self-hosted.
4. **Snapshot subprocess argv.** Mock `spawn`/`exec` to record `{ command, args, env, stdio }`; snapshot the argv for ssh/scp/aws/az/gcloud/kubectl/curl. Most P0 bugs are argv-construction regressions, and these snapshots catch them cheaply.
5. **Capture exit codes via `exitProcess`.** Spy on `exitProcess`/`process.exit` (note: `exitProcess` is bypassed under `NODE_ENV=unit`, so assert the code passed to it, or the code propagated by the command). Define and test the exit-code contract: `0` success, `1` generic error, `2` denied, `4` wait-timeout.
6. **Temp `HOME` / `P0_DIR` per test.** Point config/identity/cache/keys at a `tmp` dir so file-permission (`0600`/`0700`/`400`), merge-no-clobber, and cleanup-on-failure assertions run against real files, and so tests can't read a developer's real `~/.p0`.
7. **Fake timers for waits, streaming, retries, and propagation loops.** `vi.useFakeTimers()` + a mocked `sleep` makes the `--wait` 5-minute/10-minute races, version-check throttling, AWS/GCP propagation retries, and tunnel readiness deterministic and fast.
8. **Per-platform parameterization.** Drive `getOperatingSystem()` / `process.platform` / `isSea()` / `SHELL` from a fixture so the `cmd.exe` wrapping, RDP Windows-gate, fish `set -gx`, FIPS-on-mac-SEA, and path-location cases run as a matrix.
9. **Tag every test with its spec ID.** Prefix `it(...)` titles with `[ID]`. A failing CI line then points straight at the row here, and coverage of the spec is greppable.

## Global CLI behavior (entrypoint, parsing, help, version, errors, telemetry, FIPS)

This area covers the yargs assembly in `src/commands/index.ts` (strict mode, demandCommand, custom `.fail()`), the `src/index.ts` entrypoint (FIPS-on-mac-SEA, global crash handlers, early `loadConfig`), the version-check middleware with its ssh-proxy/ssh-resolve skip, OTEL never breaking the CLI, and the stdout-vs-stderr discipline in `src/drivers/stdio.ts`. ## Authentication & session lifecycle (login, logout, print-bearer-token)

## 15. Cross-cutting behaviors & coverage gaps

The per-command sections above test each subsystem in isolation. The cases below come from six adversarial coverage audits that look *across* the whole CLI for invariants that no single-command test establishes: the error/exit-code contract, stdout-vs-stderr discipline for machine-consumed output, signal-driven cleanup, the argument-parser's strict-vs-forwarding split, platform/packaging seams, and secret/filesystem safety. Several of these rows also flag latent bugs found during the audit (e.g. raw `sys.exit` bypassing OpenTelemetry span teardown, a non-atomic identity-file write, and FIPS-init assumptions) â€” those are called out in the relevant rows and are worth fixing alongside writing the test.

IDs in this section are prefixed `GAP-<DIM>-NN`. Each row names the **Suggested area** it belongs to so it can be co-located with the matching per-command suite if preferred.
